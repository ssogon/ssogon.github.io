인증 API

인증(authentication)이란 사용자가 누구인지 확인하는 절차다.
로그인 과정이 인증의 대표적인 예라 할 수 있다.

Form 로그인​

HttpSecurity의 formLogin()을 호출하면 기본 로그인 페이지(/login)와 Form 로그인 인증 기능이 활성화된다.

httpSecurity.formLogin()
    .loginPage("/login")
    .defaultSuccessUrl("/")
    .failureUrl("/login")
    .usernameParameter("username")
    .passwordParameter("password")
    .loginProcessingUrl("/login")
    .successHandler((request, response, authentication) -> {
      System.out.println("authentication: " + authentication.getName());
      response.sendRedirect("/");
    })
    .failureHandler((request, response, exception) -> {
      System.out.println("exception: " + exception.getMessage());
      response.sendRedirect("/login.html");
    })
    .permitAll();

formLogin()이 반환하는 FormLoginConfigurer<HttpSecurity>를 통해 Form 로그인 옵션들을 설정할 수 있다.

FormLoginConfigurer​

FormLoginConfigurer<HttpSecurity>는 Form 로그인과 관련된 설정 API를 제공한다.

메서드설명기본 값
loginPage(String)로그인 페이지를 지정한다.
이 값을 설정하면 기본 제공되는 로그인 페이지는 사라진다.		GET "/login"
defaultSuccessUrl(String)로그인 성공 후 리다이렉트할 페이지를 지정한다.
failureUrl(String)로그인 실패 후 리다이렉트할 페이지를 지정한다.GET "/login?error"
usernameParameter(String)아이디 파라미터명을 명시한다."username"
passwordParameter(String)패스워드 파라미터명을 명시한다."password"
loginProcessingUrl(String)<form> 태그의 action 속성에 들어갈 URL을 설정한다.POST "/login"
successHandler(AuthenticationSuccessHandler)로그인 성공 후 호출할 핸들러를 설정한다.
failureHandler(AuthenticationFailureHandler)로그인 실패 후 호출할 핸들러를 설정한다.
permitAll()loginPage(), failureUrl(), loginProcessingUrl()로 지정한 URL에 모든 사용자가 접근할 수 있도록 허용한다.

UsernamePasswordAuthenticationFilter​

앞서 HttpSecurity의 formLogin()으로 Form 로그인 인증 기능을 추가하고, FormLoginConfigurer<HttpSecurity>로 세부 사항들을 설정해봤다.
UsernamePasswordAuthenticationFilter는 이에 대한 실질적인 로그인 인증 처리를 수행하는 필터다.

로그인 과정​

  1. 클라이언트의 request는 UsernamePasswordAuthenticationFilter를 거치게 된다.
  2. 필터는 AntPathRequestMatcher를 통해 요청받은 URL이 loginProcessingUrl()로 지정한 값과 매칭되는지 확인한다.
    (매칭되지 않는다면 이하 과정을 생략하고 다음 필터 체인으로 넘어간다.)
  3. 클라이언트에게서 전달받은 아이디와 패스워드로 Authentication 객체를 생성한다.
  4. 생성한 Authentication은 AuthenticationManager에게 전달된다.
  5. AuthenticationManager는 AuthenticationProvider에게 인증 처리를 위임한다.
  6. AuthenticationProvider은 인증 처리를 수행한 후, 유저 정보와 권한(authorities) 정보가 담긴 Authentication를 생성해 AuthenticationManager에게 반환해준다.
    만약 인증에 실패한다면 AuthenticationException 예외가 발생한다.
  7. 최종적으로 생성된 Authentication은 UsernamePasswordAuthenticationFilter에게 반환되고, SecurityContext에 저장된다.
  8. 위 과정을 거쳐 인증에 성공한 이후에는 successHandler()로 지정한 핸들러가 호출된다.

로그아웃​

HttpSecurity의 logout()을 호출하면 기본 로그아웃 페이지(/logout)와 로그아웃 기능이 활성화된다.

httpSecurity.logout()
    .logoutUrl("/logout")
    .logoutSuccessUrl("/login")
    .deleteCookies("JSESSIONID", "remember-me")
    .addLogoutHandler((request, response, authentication) -> {
      HttpSession session = request.getSession();
      session.invalidate();
    })
    .logoutSuccessHandler((request, response, authentication) -> {
      response.sendRedirect("/login");
    });

logout()이 반환하는 LogoutConfigurer<HttpSecurity>를 통해 로그아웃 옵션들을 설정할 수 있다.

LogoutConfigurer​

LogoutConfigurer<HttpSecurity>는 로그아웃과 관련된 설정 API를 제공한다.

메서드설명기본 값
logoutUrl(String)로그아웃을 기능을 수행할 URL을 지정한다.POST "/logout"
logoutSuccessUrl(String)로그아웃 성공 후 리다이렉트할 페이지를 지정한다.GET "/login?logout"
deleteCookies(String...)로그아웃 후 삭제할 쿠키들을 지정한다.
addLogoutHandler()로그아웃시 호출할 핸들러를 설정한다.
logoutSuccessHandler()로그아웃 성공 후 호출할 핸들러를 설정한다.

LogoutFilter​

앞서 HttpSecurity의 logout()으로 로그아웃 기능을 추가하고, LogoutConfigurer<HttpSecurity>로 세부 사항들을 설정해봤다.
LogoutFilter는 이에 대한 실질적인 로그아웃 처리를 수행하는 필터다.

로그아웃 과정​

  1. 클라이언트의 request는 LogoutFilter를 거치게 된다.
  2. 필터는 AntPathRequestMatcher를 통해 요청받은 URL이 logoutUrl()로 지정한 값과 매칭되는지 확인한다.
    (매칭되지 않는다면 이하 과정을 생략하고 다음 필터 체인으로 넘어간다.)
  3. SecurityContext로부터 Authentication를 가져와 CompositeLogoutHandler에게 전달한다.
  4. CompositeLogoutHandler는 여러 핸들러를 순차적으로 호출하며 아래 로그아웃 작업들을 수행한다.
    • addLogoutHandler()로 지정한 핸들러를 호출한다.
    • deleteCookies()로 지정한 쿠키들을 삭제한다.
    • 세션을 무효화(invalidate)한다.
    • Authentication을 삭제한다.
    • SecurityContextHolder에서 SecurityContext를 삭제한다.
    • logoutSuccessHandler()로 지정한 핸들러를 호출한다.

remember-me 쿠키​

remember-me란 세션이 만료되고 웹 브라우저가 종료된 후에도 어플리케이션이 사용자를 기억하는 기능이다.
remember-me 쿠키가 request 헤더에 있다면 토큰 기반 인증을 통해 유효성을 검증하고 로그인 처리를 수행한다.

HttpSecurity의 rememberMe()를 호출하면 Form 로그인 페이지에 remember-me 체크박스가 추가되고, remember-me 인증 기능이 활성화된다.

httpSecurity.rememberMe()
    .rememberMeParameter("remember-me")
    .tokenValiditySeconds(60 * 60 * 24 * 14)
    .alwaysRemember(true)
    .userDetailsService();

rememberMe()가 반환하는 RememberMeConfigurer<HttpSecurity>를 통해 remember-me 옵션들을 설정할 수 있다.

RememberMeConfigurer​

RememberMeConfigurer<HttpSecurity>는 remember-me 기능과 관련된 설정 API를 제공한다.

메서드설명기본 값
rememberMeParameter(String)remember-me <input type="checkbox">의 name 속성에 들어갈 파라미터명을 지정한다."remember-me"
tokenValiditySeconds(int)remember-me 쿠키의 만료일을 설정한다.14일
alwaysRemember(boolean)remember-me <input type="checkbox">를 체크하지 않아도 remember-me 기능을 수행하도록 설정한다.false
userDetailsService(UserDetailsService)사용자 계정을 조회하는 서비스를 지정한다. (유효성 검증)

RememberMeAuthenticationFilter​

앞서 HttpSecurity의 rememberMe()으로 remember-me 인증 기능을 추가하고, RememberMeConfigurer<HttpSecurity>로 세부 사항들을 설정해봤다.
RememberMeAuthenticationFilter는 이에 대한 실질적인 remember-me 인증 처리를 수행하는 필터다.

remember-me 인증 과정​

최초 remember-me 로그인 시에는 아래 과정을 거친다.

  1. 클라이언트의 request는 UsernamePasswordAuthenticationFilter를 거치게 된다.
  2. 로그인에 성공했을 시 RememberMeServices의 loginSuccess()을 호출한다.
  3. RememberMeServices는 먼저 remember-me 체크박스가 체크되었는지 확인한다.
    (remember-me 체크박스가 체크되지 않았다면 이하 과정을 생략하고 다음 필터 체인으로 넘어간다.)
  4. remember-me 쿠키 발급 로직은 RememberMeServices의 구현체에 따라 달라진다.
    • NullRememberMeServices: 아무 것도 하지 않는다.
    • TokenBasedRememberMeServices: Base-64로 인코딩된 쿠키를 발급한다.
      (username + ":" + expiryTime + ":" + Md5Hex(username + ":" + expirtyTime + ":" + password + ":" + key))
    • PersistentTokenBasedRememberMeServices: 영구적인 저장소 기반으로 쿠키를 발급한다.
  5. 최종적으로 발급받은 remember-me 쿠키는 response로 내려준다.

추후 로그인 시에는 아래 과정을 거친다.

  1. 클라이언트의 request는 RememberMeAuthenticationFilter를 거치게 된다.
  2. SecurityContext로부터 Authentication를 가져와 null인지 체크한다.
    (null이 아니라면 인증을 이미 받았다는 소리이므로 이하 과정을 생략하고 다음 필터 체인으로 넘어간다.)
  3. RememberMeServices의 autoLogin()을 호출하여 사용자가 최초 로그인시 remember-me 쿠키를 발급 받았는지 확인한다.
    (remember-me 쿠키가 없다면 이하 과정을 생략하고 다음 필터 체인으로 넘어간다.)
  4. remember-me 쿠키로 부터 토큰을 추출한다.
  5. userDetailsService()로 지정한 서비스를 통해 사용자를 조회한다.
  6. 유저 정보와 권한(authorities) 정보가 담긴 RememberMeAuthenticationToken 객체를 생성하여 RememberMeAuthenticationFilter에게 반환해준다.
  7. 생성된 Authentication은 AuthenticationManager를 통해 인증 처리를 완료한 뒤 SecurityContext에 저장되게 된다.

익명 사용자​

익명 사용자란 인증을 받지 않은 사용자를 null로 처리하지 않고 별도의 Authentication 객체를 만들어 분류하겠다는 전략이다.
스프링은 이처럼 null을 피하기 위해 Null Object Pattern을 사용하는 모습을 자주 보인다.

Authentication authentication = SecurityContextHolder.getContext().getAuthentication();
if (authentication instanceof AnonymousAuthenticationToken) {
  // show login button
}
if (authentication.isAuthenticated()) {
  // show logout button
}

AnonymousAuthenticationFilter​

익명사용자를 처리하는 AnonymousAuthenticationFilter 필터는 DefaultSecurityFilterChain에 들어가는 기본 필터이기 때문에 별다른 설정을 하지 않아도 적용된다.

익명 사용자 Authentication 생성 과정​

  1. 클라이언트의 request는 AnonymousAuthenticationFilter를 거치게 된다.
  2. SecurityContext로부터 Authentication를 가져와 null인지 체크한다.
    (null이 아니라면 인증을 이미 받았다는 소리이므로 이하 과정을 생략하고 다음 필터 체인으로 넘어간다.)
  3. 익명 사용자용 Authentication 객체인 AnonymousAuthenticationToken을 생성한다.
    (principal="anonymousUser", authorities=["ROLE_ANONYMOUS"])
  4. 생성된 Authentication은 SecurityContext로 전달된다.

세션 설정​

HttpSecurity의 sessionManagement()가 반환하는 SessionManagementConfigurer<HttpSecurity>를 통해 세션과 관련된 옵션들을 설정할 수 있다.

httpSecurity.sessionManagement()
    .invalidSessionUrl("/invalid")
    .expiredUrl("/expired")
메서드설명기본 값
invalidSessionUrl(String)세션이 유효하지 않을 때 리다이렉트할 페이지를 지정한다.
expiredUrl(String)세션이 만료되었을 때 리다이렉트할 페이지를 지정한다.

동시 세션 제어​

동시 세션 제어란 동일 계정의 최대 허용 세션 수가 초과되었을 경우 어떻게 처리할 것인가 결정하는 전략이다.
스프링 시큐리티에서는 두 가지 전략으로 동시 세션을 제어한다.

  • 이전 사용자의 세션을 만료시키는 전략
  • 이후 사용자의 인증은 실패하도록 하여 로그인을 차단시키는 전략
httpSecurity.sessionManagement()
    .maximumSessions(1)
    .maxSessionsPreventsLogin(true)
메서드설명기본 값
maximumSession(int)최대 허용 가능한 세션 수를 지정한다.
-1은 무제한을 나타낸다.		1

세부 사항은 maximumSession(int)이 반환하는 ConcurrencyControlConfigurer를 통해 설정할 수 있다.

메서드설명기본 값
maxSessionPreventsLogin(boolean)최대 허용 가능 세션 수를 초과했을 경우 이후 사용자의 인증은 실패하도록 하는 전략을 사용할 것인지 설정한다.false
(이전 사용자의 세션을 만료시키는 전략)

세션 고정(session fixation) 보호​

세션 고정(session fixation)이란 공격자가 자신의 세션 쿠키를 공격할 대상의 브라우저에 심어 놓고 그 대상이 심어놓은 쿠키로 로그인을 하면 결과적으로 같은 세션을 공유하게 되는 보안 공격의 한 종류다.
세션 고정 보호는 이러한 세션 고정 공격을 방지하기 위해 인증할 때마다 새로운 세션을 생성해 공격자가 심어 놓은 쿠키는 의미가 없어지도록 만들어버리는 전략이다.

httpSecurity.sessionManagement()
    .sessionFixation()
    .changeSessionId();
메서드설명
sessionFixation()세션 고정 보호 기능을 활성화한다.

세션 고정 보호 정책은 sessionFixation()이 반환하는 SessionManagementConfigurer를 통해 설정할 수 있다.

메서드설명
changeSessionId()사용자가 인증에 성공하면 세션 아이디를 변경한다.
서블릿 3.1 이상에서 작동한다.
newSession()사용자가 인증에 성공하면 새로운 세션을 생성해 발급한다.
이전 세션에 담긴 정보는 활용하지 못한다.
migrateSession()사용자가 인증에 성공하면 세션 아이디를 변경한다.
서블릿 3.0 이하에서 작동한다.
none()세션 고정 보호 기능을 사용하지 않는다.

세션 고정 보호 기능은 별도로 설정하지 않아도 서블릿 버전에 따라 changeSessionId() 또는 migrateSession()이 적용된다.

세션 생성 정책​

세션 생성 정책이란 스프링 시큐리티가 세션을 어떻게 생성하고 관리할 것인지 결정하는 정책이다.

httpSecurity.sessionManagement()
    .sessionCreationPolicy(SessionCreationPolicy.If_Required)
메서드설명
sessionCreationPolicy(SessionCreationPolicy)세션 정책을 설정한다.

세션 정책을 나타내는 열거형 SessionCreationPolicy가 가질 수 있는 값은 다음과 같다.

  • SessionCreationPolicy.Always: 스프링 시큐리티가 항상 세션을 생성한다.
  • SessionCreationPolicy.If_Required: 스프링 시큐리티가 필요시 세션을 생성한다. (기본 값)
  • SessionCreationPolicy.Never: 스프링 시큐리티가 세션을 생성하지 않지만, 사용은 한다.
  • SessionCreationPolicy.Stateless: 스프링 시큐리티가 세션을 생성하지 않으며, 사용하지도 않는다.

SessionManagementFilter​

SessionManagementFilter는 위에서 살펴본 세션 관리 및 동시 세션 제어, 세션 고정 보호, 세션 생성 정책을 수행하는 필터다.

ConcurrentSessionFilter​

ConcurrentSessionFilter는 동시 세션 제어에 관여하는 필터로, 매 요청마다 사용자의 세션 만료 여부를 체크하는 필터다.
세션이 만료되었을 경우에는 해당 사용자를 로그아웃 처리한 후 오류 페이지로 리다이렉트시켜준다.

동시 세션 제어 과정​

  1. 클라이언트의 request는 SessionManagementFilter를 거치게 된다.
  2. SessionManagementFilter는 아래 3개의 클래스를 차례대로 호출한다.
    1. ConcurrentSessionControlAuthenticationStrategy는 동시적 세션 제어를 처리하는 클래스다.
      현재 인증을 시도하고 있는 사용자의 session count가 최대 세션 허용 수를 초과하는지 확인하는 역할을 한다.
      최대 세션 허용 수가 초과되었을 경우에는 설정한 전략에 따라 다르게 동작한다.
      • 세션 만료 전략인 경우 이전 사용자의 SessionInformation를 expireNow()로 만료시킨다.
      • 인증 실패 전략인 경우 SessionAuthenticationException을 발생시켜 새로운 사용자의 로그인을 막는다.
    2. ChangeSessionIdAuthenticationStrategy는 세션 고정 보호 처리를 하는 클래스다.
      설정한 세션 고정 보호 정책에 따라 세션 아이디를 매번 변경하거나 새로운 세션을 생성하는 등의 작업을 한다.
    3. RegisterSessionAuthenticationStrategy는 사용자의 세션을 등록하고 저장하는 클래스다.
      이 클래스에 의해 세션이 등록된 후에야 session count가 증가한다.
  3. 클라이언트의 request는 ConcurrentSessionFilter도 거친다.
  4. ConcurrentSessionFilter에서는 SessionInformation의 isExpired()를 통해 현재 세션이 만료되었는지 확인한다.
  5. 세션 만료 전략에 의해 만료된 사용자는 로그아웃된 후 오류 페이지로 리다이렉트된다.
