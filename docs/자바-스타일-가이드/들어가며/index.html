들어가며

이 가이드의 목적은 보다 좋은 코드를 장려하기 위한 일련의 규칙을 제공하는 것이다.
만약 이미 따르고 있는 팀 컨벤션이 있다면 그 규칙을 계속해서 따라야 한다.
이 문서는 마땅히 따르는 컨벤션이 없는 이들을 위해 여러 가이드를 한 데 모아 정리해둔 것뿐이다.

여기서 소개할 대부분의 규칙은 Oracle Code Conventions, Google Style Guide, Twitter Style Guide 등의 코딩 스타일 가이드나 SonarSource의 정적 분석 규칙을 의역한 것이며 클린 코드, 이펙티브 자바, 리팩터링 번역본 내용을 일부 인용하기도 하였다.

여러 문서와 책을 참조하여 작성하였으나 정형화가 거의 완료되어있어 서로 상충하는 일은 드물었다.
각 항목마다 출처를 기재해두었으니 참고하면 도움이 될 것이다.

코드 분석 도구​

코딩 컨벤션을 숙지하고 있는 것도 좋지만 자신이 사용하는 IDE에 SonarLint와 같은 코드 분석 플러그인을 설치하여 코드를 수시로 검증받는 것이 좋다.
정적 분석 플러그인을 사용하면 사설망 같은 오프라인 환경에서도 코드를 점검받을 수 있다.

클린 코드​

클린 코드는 깨끗한 코드를 만드는 애자일 기법들을 정제해 담은 책이다.
SOLID 원칙으로도 유명한 Robert Martin이 집필하였다.

  • 클린 코드 (번역본)

프로그래머라면 각자 선호하는 규칙이 있다.
하지만 팀에 속한다면 자신이 선호해야 할 규칙은 바로 팀 규칙이다.
팀은 한 가지 규칙에 합의해야 한다. 그리고 그 모든 팀원은 그 규칙을 따라야 한다.
그래야 소프트웨어가 일관적인 스타일을 보인다.
개개인이 따로국밥처럼 맘대로 짜대는 코드는 피해야 한다.

좋은 소프트웨어 시스템은 읽기 쉬운 문서로 이뤄진다는 사실을 기억하기 바란다.
스타일은 일관적이고 매끄러워야 한다.
한 소스 파일에서 봤던 형식이 다른 소스 파일에도 쓰이리라는 신뢰감을 독자에게 줘야 한다.
온갖 스타일을 뒤섞어 소스 코드를 필요 이상으로 복잡하게 만드는 실수는 반드시 피한다.

- 클린 코드 (번역본) 113, 114쪽에서

이펙티브 자바​

이펙티브 자바는 자바의 설계와 구현을 주도하였던 Joshua Bloch의 개발 철학이 담긴 책이다.
되도록이면 개정 3판을 보는 것을 추천한다. 자바 8에 관한 내용이 추가되었다.

  • 이펙티브 자바 (번역본)

나는 기본적으로 공개된 API 관점에서 생각하며, 여러분도 그렇게 하기를 권한다.
재사용 컴포넌트를 개발하지 않을 때라도, 이 방식이 여러분이 작성하는 소프트웨어 품질을 끌어올려 줄 것이다.

익스트림 프로그래밍(XP) 같은 경량 개발 방법론을 선호하는 개발자에게는 API 설계에 대한 내 집착이 조금 과하다 느껴질지도 모르겠다.
이들 방법론에서는 동작하는 가장 단순한 프로그램을 강조한다.
경량 방법론을 따르는 독자라면 API 설계가 리팩터링 과정에 큰 도움이 된다는 걸 발견할 것이다.
리팩터링의 핵심 목표인 시스템 구조 개선과 중복 제거는 잘 설계된 API를 건너뛰고는 도달할 수 없기 때문이다.

- 이펙티브 자바 (번역본) 서문에서

리팩터링​

리팩터링은 객체지향 설계의 선구자인 Martin Fowler의 책으로, 다양한 리팩토링 기법들을 소개한다.
2판부터는 자바스크립트로 작성되어 있다는게 약간 흠이지만, ES6 문법까지만 알고 있다면 읽는데 큰 지장은 없을 것이다.

  • 리팩터링 (번역본)

컴퓨터가 이해하는 코드는 어느 바보나 짤 수 있다.
좋은 프로그래머는 사람이 이해하는 코드를 짠다.

- Martin Fowler
