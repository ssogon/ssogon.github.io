네이밍 컨벤션

팀 구성원간의 협의로 표준화된 네이밍 규칙은 상대의 코드를 더 읽기 쉽게 만들어준다.

공통 규칙​

의도 나타내기​

좋은 이름은 표현하는 대상에 대한 정보를 내포하여 코드를 이해하는데 도움을 준다.
따로 주석이 필요하다면 그 의도를 분명히 드러내지 못했다는 뜻이다.

int s; // bad
int elapsedSeconds; // good

좋은 이름을 짓는 일은 많은 시간을 할애해야 하는 작업이지만 좋은 이름으로 절약하는 시간이 훨씬 많다.
더 나은 이름이 떠오른다면 나중에라도 개선하자.

클린 코드: 의도를 분명히 밝혀라 (22page)

발음하기 쉬운 이름​

사람은 말하는 것에 능숙하며 우리 두뇌의 상당 부분은 단어의 개념에만 전념한다.
말을 처리하려고 발달한 두뇌를 활용하지 않는 것은 손해다.

String lstUpdYyyymmddhhmiss; // bad: 팀원들 간에 언급해야 하는 상황을 상상해보라.
String lastUpdateDateTime; // good: 위의 예보다 지적인 대화가 가능해진다.

클린 코드: 발음하기 쉬운 이름을 사용하라 (27page)

잘 알려진 약어​

개발자들이 잘 알고 있는 약어의 경우 충분히 유추 가능하기 때문에 사용해도 문제되지 않는다.

Config config; // configuration
String id; // identifier
Spec spec; // specification
Statistics stats; // statistics

다만 아래처럼 약어가 겹치거나 파편화되는 경우는 반드시 피해야한다.

Result res;
Response res;
Response resp;

Google API Design Guide: Name abbreviation

낙타 표기법과 약어​

낙타 표기법을 사용하는 네이밍 컨벤션은 약어가 포함되어 있더라도 낙타 표기법을 고수하여 네이밍을 일관되게 가져가야 한다.
개발자에게 고민할 여지를 주지 말자.

class JSONObject { ... } // bad
class JsonObject { ... } // good

Google Style Guide: Camel case

패키지명​

패키지명은 lower case로 작성한다.
하이픈(-)이나 언더스코어(_)는 포함되지 않는다.

package com.example.service;

버전이 지정된 API 패키지명은 해당 버전으로 끝나야 한다.

package com.example.service.v2;

Oracle Code Conventions: Naming Conventaions
Google Style Guide: Package names
Google API Design Guide: Package names
SonarSource: Package names should comply with a naming convention

클래스명​

클래스명은 upper camel case로 작성한다.
통상적으로 명사나 명사구를 사용한다.

class ImmutableList { ... }

클린 코드: 클래스 이름 (32page)
Oracle Code Conventions: Naming Conventaions
Google Style Guide: Class names
SonarSource: Class names should comply with a naming convention

추상클래스명​

추상클래스명은 Abstract로 시작하는 것이 관례다.

abstract class AbstractMap { ... }

SonarSource: Abstract class names should comply with a naming convention

인터페이스명​

인터페이스명은 명사나 명사구가 될 수 있고, 형용사나 형용사구가 될 수도 있다.

interface List { ... }
interface Iterable { ... }

간혹 인터페이스명에 접두사 I를 붙이는 경우가 있는데 이는 인터페이스임을 나타낼 수는 있겠지만 객체지향적이지 않은 명명법이니 지양하는 것이 좋다.

interface IService { ... } // bad
interface Serviceable { ... } // good

Oracle Code Conventions: Naming Conventaions
Google Style Guide: Class names
SonarSource: Interface names should comply with a naming convention

메서드명​

메서드명은 lower camel case로 작성한다.
통상적으로 동사나 동사구로 시작한다.

void fileDownload() { ... } // bad: 명사로 시작해선 안된다.
void downloadFile() { ... } // good

메서드명은 동작하는 바를 명확히 표현할 수 있어야 한다.

Member fetchData(long id) { ... } // bad: fetch 대상이 도대체 무엇인가?
Member fetchMember(long id) { ... } // good

도메인 객체의 메서드는 직설법(indicative mood)을, API의 메서드는 명령법(imperative mood)을 사용하는 것이 자연스럽다.

class Book {
  boolean isBookPublisherApproved() { ... }
}
class BookService {
  boolean checkBookPublisherApproved(Book book) { ... }
}

메서드명에 For, With, At, To, From 등의 전치사를 붙이면 자칫 장황해보일 수 있어 피하는 것이 좋다.
일반적으로 전치사가 포함된 메서드는 인자를 추가하거나 별도의 동명사를 다는 식으로 회피할 수 있다.

Oracle Code Conventions: Naming Conventaions
Google Style Guide: Method names

SonarSource: Method names should comply with a naming convention

boolean을 반환하는 메서드명​

반환 값이 boolean인 메서드명은 is, has, 조동사를 적절히 사용하여 논리 값이 반환됨을 내포할 수 있어야 한다.

boolean isRunning() { ... }
boolean isFinished() { ... }
boolean isEachThreadFinished() { ... }
boolean hasToken() { ... }
boolean canCancel() { ... }

부정형은 not(!) 연산시 가독성이 떨어진다는 문제가 있어 항상 긍정형을 사용해야 한다.

if (!account.isDisabled()) { ... } // bad: '비활성화되지 않은', 가독성이 떨어진다.
if (!account.isEnabled()) { ... } // good: '활성화되지 않은', 읽기 쉽다.

boolean 메서드의 스타일은 boolean 변수에도 동일하게 적용된다.

SonarSource: The names of methods with boolean return values should start with "is" or "has"
토막글: Tips on naming boolean variables - Cleaner Code

콜백 메서드명​

특정 이벤트가 발생했을 시 호출되는 콜백 메서드는 보통 on 또는 handle로 시작하는 것이 관례다.

void onClick() { ... }
void handleClick() { ... }

변수명​

상수가 아닌 변수명​

상수가 아닌 변수명은 lower camel case로 작성한다.
통상적으로 명사 또는 명사구를 사용한다.

String name = "scott";

대상이 복수인 경우에는 이를 적절히 표현해주어야 한다.

List<String> names = Arrays.asList("scott", "holden");

Oracle Code Conventions: Naming Conventaions
Google Style Guide: Non-constant field names
Google API Design Guide: Collection IDs
SonarSource: Local variable and method parameter names should comply with a naming convention

상수명​

상수명은 upper underscore case로 작성한다.

static final String NAME = "scott";
static final ImmutableList<String> NAMES = ImmutableList.of("scott", "holden");
static final Joiner COMMA_JOINER = Joiner.on(','); // Joiner는 불변 클래스다.
static final String[] EMPTY_ARRAY = {};
enum CharSet { EUC_KR, UTF_8 };

static 또는 final 키워드가 없다면 상수가 아니다.

static String name = "scott";
final int age = 20;

변경 가능한 객체 또한 상수가 아니다.

static final List<String> names = Arrays.asList("scott", "holden");
static final Logger logger = Logger.getLogger(Service.getClass());
static final String[] nonEmptyArray = {"these", "can", "change"};

상수의 네이밍을 이처럼 특별하게 함으로써 얻는 이점은 개발자가 변수의 선언부를 굳이 확인하지 않더라도 해당 변수가 불변하다는 것을 알 수 있게 해주어 코드 트래킹 반경을 줄여준다는 것에 있다.

Oracle Code Conventions: Naming Conventaions
Google Style Guide: Constant names
SonarSource: Constant names should comply with a naming convention
SonarSource: Local constants should follow naming conventions for constants

너무 긴 변수명​

변수명에 의도를 잘 담아내려고 하다 보면 자칫 변수명이 길어질 수도 있다.
그렇다고 단어를 줄여쓰는 것은 긴 것만도 못한 결과를 야기시킨다.

String postWriterFirstName = "scott"; // bad: 너무 길다.
String pstWrtrFstNm = "scott"; // bad: 가독성이 떨어질뿐더러 의미를 파악하기 힘들다.

변수명이 길어졌다는 것은 클래스로 개념을 분리할 필요가 있다는 의미다.

Member member = new Member();
member.setFirstName("scott");

Board board = new Board();
board.setWriter(member);

클린 코드: 불필요한 맥락을 없애라 (37page)

루프 인덱스​

루프문의 인덱스 변수는 전통적으로 한 글자를 사용해왔기 때문에 i, j, k처럼 짧은 이름은 크게 문제되지 않는다.

for (int i = 0; i < names.length; i++) {
  logger.info(names[i]);
}

하지만 루프의 범위가 크고 복잡하다면 의미있는 이름을 고려해볼 필요가 있다.

for (int nameIndex = 0; nameIndex < names.length; nameIndex++) {
  // do many things...
  logger.info(names[nameIndex]);
}

클린 코드: 자신의 기억력을 자랑하지 마라 (31page)

매직 넘버​

매직 넘버는 뜬금 없이 등장하는 숫자 리터럴을 일컬으며, 보통 루프의 반복횟수에 자주 등장한다.
이러한 매직 넘버는 간단해보일지라도 디버깅을 어렵게 만드는 주범이다.

for (int i = 0; i < 4; i++); // bad
for (int i = 0; i < NUMBER_OF_CYCLES; i++); // good

다만 -1, 0, 1은 매직 넘버로 간주되지 않는다.

SonarSource: Magic numbers should not be used

시간과 기간​

시간을 나타낼 때는 동사에 과거시제를 사용하지 않는다.

LocalDateTime lastUpdatedTime; // bad
LocalDateTime lastUpdateTime; // good

정수 형식을 사용해 타임스탬프를 나타내야 할 경우에는 변수명에 단위 접미사를 포함시킨다.

int sendTimeMillis = 1000; // xxx{Time|Duration|Delay|Latency|...}{Seconds|Millis|Micros|Nanos|...}

문자열 형식을 사용해 타임스탬프를 나타내는 경우에는 단위 접미사를 포함시키지 않는다.

String sendTime = "2014-07-30T10:43:17Z";

Google API Design Guide: Time and Duration

수량​

정수 형식의 수량에는 단위가 포함되어야 한다.
수량이 항목 개수인 경우에는 count 접미사를 사용한다.

int nodeCount = 1; // xxx{Bytes|WidthPixels|Meters|...}

Google API Design Guide: Quantities
Twitter Style Guide: Include units in variable names

메타 데이터​

흔히들 하는 실수는 변수명에 메타 데이터를 포함시키는 것이다.
변수명은 그 변수의 목적을 표현하는 데에만 포커스를 맞춰야 한다.

타입 정보​

이름에 타입 정보를 표시해야 한다는 것은 변수의 타입을 곧 바로 확인할 수 없을 정도로 변수를 선언한 위치와 사용하는 위치가 멀다는 이야기 밖에 되지 않는다.

String nameString = "scott"; // bad: 타입에 대한 정보는 필요없다.
String name = "scott"; // good

스코프 정보​

멤버 변수에 m이나 _ 같은 접두어를 붙일 필요도 없다.
클래스와 메서드는 접두어가 필요없을 정도로 작아야 마땅하다.

int mAge = 10000; // bad: 스코프에 대한 정보는 필요없다.
int age = 10000; // good

불용어​

Data나 Info와 같은 단어는 a, an, the와 마찬가지로 아무런 정보도 제공하지 못한다.
더군다나 memberData와 memberInfo처럼 두 불용어를 혼용한다면 검색하기만 힘들어질 것이다.

Member memberData = new Member(); // bad: 불용어는 필요없다.
Member member = new Member(); // good

다만 명사가 아닌 동사 뒤에 붙는다면 다른 의미로 다가올 수 있다.

PlayInfo playInfo = new PlayInfo(); // not bad

클린 코드: 인코딩을 피하라 (29page)
Twitter Style Guide: Don't embed metadata in variable names

마치면서​

많은 이들은 다른 개발자가 반대할 것을 우려하여 이름 바꾸는 것을 두려워한다.
좋은 이름의 힘을 아는 사람은 이름이 개선되었을 때 고마움을 느끼겠지만, 코드가 바뀌었다며 질책하는 사람도 있을지 모른다.

하지만 그것이 코드를 개선하려는 노력을 중단하는 이유가 되어서는 안된다.
정진하다 보면 단기적인 효과는 물론 장기적인 성과도 따라올 것이다.
