코드 포맷팅

코드 포맷팅은 의사소통의 일환이자 개발자의 일차적인 의무다.
규칙적이며 획일화된 코드 양식은 가독성 수준을 높여준다.

클린 코드: 형식을 맞추는 목적 (96page)

고정폭 글꼴​

고정폭(monospace) 글꼴이란 각 글자가 동일한 너비를 가지는 글꼴을 말한다.
너비가 일정하기 때문에 오탈자를 찾기 쉽고, 블록 구조의 코드를 빠르게 이해할 수 있다.

한글의 고정폭까지 지원하는 D2Coding 사용을 권장한다.

Wikipedia: Monospaced font

협업을 위한 IDE 설정​

들여쓰기​

새로운 블록이 열릴 때 들여쓰기를 어떻게 하느냐는 개발자들의 영원한 논쟁거리다.

들여쓰기설명
2-space간결하지만 깊이 중첩된 코드는 자칫 혼잡해보일 수 있다.
하지만 이는 모듈 분리가 필요하다는 뜻이기도 하다.
4-space깊이 중첩된 코드도 읽는데 큰 어려움이 없다.
2-space 파벌은 별 차이가 없다고 주장한다.
tab개인이 원하는 대로 들여쓰기 크기를 설정해도 협업 시 문제가 되지 않는다.
다만 라인 래핑시 탭 문자만으로는 유연한 들여쓰기가 불가능해 공백을 혼용하곤 하는데 이 경우 개발자의 들여쓰기 크기 설정에 따라 코드가 엉망으로 보이게 될 수 있다.

참고로 구글과 트위터는 2-space를, 오라클은 4-space를 권장하고 있다.

tab은 장점보다 단점이 더 커서 점점 사용하지 않는 추세다.
최신 IDE에는 탭 키를 누를 시 공백으로 처리해주는 기능이 있으니 이를 활용하자.

중괄호 스타일​

중괄호 스타일의 경우 K&R 스타일을 변형한 1TBS를 사용하는 것으로 정형화 되어 있다.

Member fetchMember(long id) {
  Member member = repository.find(Member.class, id);
  return member;
}

중괄호 안의 내용이 한 줄이라 하더라도 가독성을 위해 줄바꿈을 생략하지 않는다.
가독성에 크게 해가 되지 않는다 생각할 수도 있지만 최신 IDE에는 이러한 블록을 한 줄로 보여주는 기능도 있으니 개발자가 직접 줄일 필요는 없다.

String getName() {
  return this.name;
}

빈 블록의 경우 줄바꿈을 생략해도 문제되지 않는다.

void doSomething() {}

다만 다중 블록의 경우에는 빈 블록이라 하더라도 줄바꿈을 생략하지 않는다.

try {
  doSomething();
} catch (Exception ignored) {} // bad

공백​

키워드와 소괄호, 중괄호 사이에는 공백을 넣어준다.

if (isActive) {
  logger.info("active");
} else {
  logger.info("inactive");
}

연산자의 앞 뒤에도 공백을 넣어준다.

int sum = 0;
for (int i = 1; i < 50; i++) {
  sum += i;
}

Google Style Guide: Horizontal whitespace
Twitter Style Guide: Space pad operators and equals

라인 래핑​

한 라인이 길어지는 경우 가독성을 위해 줄바꿈을 할 수 있다.
이 경우 기존 들여쓰기의 2배만큼 들여쓰는 것이 일반적이다.

List<String> names = Streams.of("scott", "holden")
    .map(String::toUpperCase)
    .collect(Collectors.toList());

일반적으로 권장하는 한 라인의 상한선은 80 또는 120 문자가 보편적이다.
고정폭 폰트를 사용하고 IDE의 상한선 기능을 사용하면 이를 쉽게 파악할 수 있다.

가독성 향상​

빈 줄​

논리적인 그룹을 나타내기 위해 빈 줄은 필요할 수 있다.
그러나 연속적인 빈 줄은 불필요하다.

수평 정렬​

수평 정렬이란 선언문, 할당문, 주석 등을 테이블처럼 정렬하는 것을 말한다.

// 수평 정렬의 예
String name   = "scott";     // 이름
int    age    = 20;          // 나이
Gender gender = Gender.MALE; // 성별

수평 정렬은 가독성에 도움이 될 수도 있겠지만 향후 유지관리에 문제가 될 가능성이 높다.
개발자는 더 자주 공백을 조정하게 되고, 계속해서 버전관리 기록을 손상시키게 될 것이다.

정렬이 필요할 정도로 가독성이 나쁘다는 것은 모듈을 분리해야 한다는 것을 뜻하기도 한다.

특정 구문​

클래스 구조​

클래스의 필드와 메서드는 다음 순서대로 작성하도록 한다.

public static PUBLIC_CONSTANT;
private static PRIVATE_CONSTANT;
private variable;
public void method() { ... }

private 메서드는 자신을 호출하는 public 메서드 뒤에 오도록 하여 신문 기사처럼 쉽게 읽히도록 하자.

Clean Code: 클래스 체계 (172p)

수식어 순서​

수식어는 다음 순서대로 작성한다.

public protected private abstract default static final transient volatile synchronized native strictfp

Google Style Guide: Modifiers
SonarSource: Modifiers should be declared in the correct order

배열 스타일​

배열의 대괄호를 c언어 스타일로 작성해서는 안된다.

String args[]; // bad
String[] args; // good

Google Style Guide: No C-style array declarations
SonarSource: Array designators "[]" should be on the type, not the variable

마치면서​

코드 포맷팅은 다양한 바리에이션이 존재하고 아직까지 정형화되지 않은 부분이 많다.
팀원들과 충분히 상의한 끝에 합의점을 찾는 것이 중요하겠다.
