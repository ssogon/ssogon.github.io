프로그래밍 관례

여기서는 올바른 코드를 위해 권장되는 여러 프로그래밍 관례들을 나열한다.
이는 선대 개발자들이 수년 간의 수행착오 끝에 도달한 경험의 집대성이라 볼 수 있다.

클래스​

단일 책임 원칙​

SOLID 원칙으로 유명한 단일 책임 원칙(SRP, Single Responsibility Principle)은 각 클래스가 하나의 책임과 목적만을 가져야 한다는 원칙이다.
클래스를 하나의 관심사에만 집중하게 만듦으로써, 클래스 간의 결합도를 낮추는 효과를 기대할 수 있다.

Baeldung: A Solid Guide to SOLID Principles
Baeldung: Single Responsibility Principle in Java

유틸리티 클래스​

정적 메서드만 있는 유틸리티 클래스를 인스터스화해서는 안된다.
private 생성자를 통해 생성될 여지가 없도록 만들자.

public class TextUtils {
  private TextUtils() {}
}

이펙티브 자바 3/E: 인스턴스화를 막으려거든 private 생성자를 사용하라 (26page)
SonarSource: Utility classes should not have public constructors
SonarSource: Classes with only "static" methods should not be instantiated

내부 클래스​

비정적 내부 클래스는 외부 클래스에 대한 참조를 가지고 있다.
만약 내부 클래스가 외부 클래스에 대한 참조가 필요하지 않은 경우에는 정적으로 선언하는 것이 메모리 측면에서 더 효율적이다.

public class Fruit {
  public static class Seed {}
}

이펙티브 자바 3/E: 멤버 클래스는 되도록 static으로 만들라 (146page)
SonarSource: Inner classes which do not reference their owning classes should be "static"

인터페이스 사용​

인터페이스는 구현에서 기능을 분리하여 사용처의 변경 없이 여러 구현을 가질 수 있게 만들어준다.
만약 인터페이스로 인해 소스 파일이 여러 개 생기는 것이 고민이라면 아래 패턴을 고려해보자.

interface FileFetcher {
  File getFile(String name);

  // All the benefits of an interface, with little source management overhead.
  // This is particularly useful when you only expect one implementation of an interface.
  static class HdfsFileFetcher implements FileFetcher {
    @Override File getFile(String name) {
      ...
    }
  }
}

Twitter Style Guide: Use interfaces

의존성 주입​

의존성 주입(dependency injection)이란 생성자 또는 setter를 통해 필요한 의존성을 외부로 부터 제공받는 패턴을 말한다.
이때 의존성을 인터페이스로 받음으로써 클래스 구성을 보다 유연하게 가져갈 수 있다.

public class Controller {
  private final Service service;

  public Controller(Service service) {
    this.service = service;
  }
}

의존성 주입이 유연성과 테스트 용이성을 개선해주기는 하지만 큰 프로젝트에서는 코드를 어지럽게 만들기도 한다.
대거(Dagger), 주스(Guice), 스프링(Spring) 같은 의존성 주입 프레임워크를 사용하면 이런 어지러움을 해소할 수 있다.

스프링의 경우 어노테이션 또는 외부 매핑 파일을 통해 주입할 대상을 결정함으로써 재컴파일 없이 시스템을 재구성할 수도 있다.

이펙티브 자바 3/E: 자원을 직접 명시하지 말고 의존 객체 주입을 사용하라 (28page)

필드 쉐도잉​

자식 클래스의 필드가 부모 클래스의 필드를 가리는 행위는 개발자를 혼란스럽게 만든다.

public class Fruit {
  protected Season ripe;
}
public class Raspberry extends Fruit {
  private boolean ripe; // bad
}

SonarSource: Child class fields should not shadow parent class fields

불변성 선호​

상수와 불변 객체는 중간에 데이터가 변경될 수 있다는 부담을 주지 않는다.
개발자는 선언부만 잘 확인하면 되기 때문에 코드를 읽는데 소요하는 시간이 크게 단축된다.

Twitter Style Guide: Favor immutability
Twitter Style Guide: Use final fields

thread-safe​

객체가 불변하다는 것은 재사용해도 thread-safe하다는 것과 같다.
Pattern처럼 생성 비용이 비싼 객체는 한 번만 생성해놓고 재사용하자.

public class RomanNumerals {
  private static final Pattern ROMAN = Pattern.compile(
    "^(?=.)M*(C[MD]|D?C{0,3})(X[CL]|L?X{0,3})(I[XV]|V?I{0,3})$");

  public static boolean isRomanNumeral(String string) {
    return ROMAN.mather(string).matches();
  }
}

그런데 예제로 든 RomanNumerals 클래스가 초기화된 후 한 번도 사용되지 않는다면 ROMAN 필드는 쓸데없이 초기화된 꼴이 된다.
isRomanNumeral(String string)이 처음 호출될 때 필드를 초기화하는 지연 초기화(lazy initialization)로 불필요한 초기화를 없앨 수는 있지만 코드를 복잡하게 만들 뿐 성능은 크게 개선되지 않으므로 권장하진 않는다.

이펙티브 자바 3/E: 불필요한 객체 생성을 피하라 (31page)

메서드​

정적 팩토리 메서드​

생성자를 오버라이딩하는 것보다는 정적 팩터리 메서드를 정의하는 것이 가독성을 높이는 데 도움이 된다.

Complex fulcrumPoint = Complex.fromRealNumber(23.0);

정적 팩터리 메서드에 일반적으로 사용되는 명명 방식은 다음과 같다.

  • from: 매개변수를 하나 받아서 해당 타입의 인스턴스를 반환한다.
  • of: 여러 매개변수를 받아 적절한 타입의 인스턴스를 반환한다.
  • valueOf: from과 of의 더 자세한 버전이다.
  • instance, getInstance: (매개변수를 받는다면) 매개변수로 명시한 인스턴스를 반환하지만, 같은 인스턴스임을 보장하지는 않는다.
  • create, newInstance: instance, getInstance 같지만, 매번 새로운 인스턴스를 생성해 반환함을 보장한다.
  • get{Type}: getInstance와 같으나 생성할 클래스가 아닌 다른 클래스에 팩터리 메서드를 정의한다.
    여기서 {Type}은 팩터리 메서드가 반환할 객체 타입의 이름이다.
  • new{Type}: newInstance와 같으나 생성할 클래스가 아닌 다른 클래스에 팩터리 메서드를 정의한다.
    여기서 {Type}은 팩터리 메서드가 반환할 객체 타입의 이름이다.
  • {type}: getType, newType의 간결한 버전이다.

클린 코드: 메서드 이름 (32page)
이펙티브 자바 3/E: 생성자 대신 정적 팩터리 메서드를 고려하라 (8page)

매개변수 타입​

메서드의 재사용성을 극대화하기 위해 매개변수는 가능한 한 상위 유형이 되어야 한다.

void logSize(List<String> list) { // bad
  logger.info(list.size());
}
void logSize(Collection<?> collection) { // good
  logger.info(collection.size());
}

SonarSource: Method parameters should be declared with base types

매개변수 개수​

가장 이상적인 매개변수의 개수는 0개로, 적을 수록 좋다.
매개변수는 개발자에게 그 의미를 유추하게 만들고 자바독을 들여다보게 만든다.

void render(boolean isAdmin) { ... } // bad: render(true)는 헷갈리기 십상이다.
void renderForAdmin() { ... } // good

특히나 render(boolean)처럼 선택자 인수를 두는 경우는 더더욱 그 의미를 유추하기 힘들게 만드므로 반드시 피해야 한다.

클린 코드: 함수 인수 (50page)
SonarSource: Public methods should not contain selector arguments

출력 값을 담는 매개변수​

일반적으로 매개변수는 함수의 입력으로 생각되기 때문에 출력 값을 담는 용도로 사용되는 매개변수는 인지적으로 거슬릴 수 밖에 없다.

int[] src = { 1, 2, 3 };
int[] dest = new int[src.length];
System.arrayCopy(src, 0, dest, 0, src.length); // bad

클린 코드: 출력 인수 (56page)

컬렉션을 반환하는 메서드​

배열이나 컬렉션을 반환하는 메서드는 null 대신 빈 객체를 반환해야 한다.
호출하는 측에서 null이 반환되는 경우를 고려하지 않도록 하자.

List<Member> fetchMembers() {
  List<Member> members = repository.find(Member.class);
  if (members == null) {
    return Collections.emptyList();
  }
  return members;
}

이펙티브 자바 3/E: null이 아닌, 빈 컬렉션이나 배열을 반환하라 (323page)
SonarSource: Empty arrays and collections should be returned instead of null

즉시 반환되는 변수​

즉시 반환될 변수를 선언해두는 것은 그리 좋지 않은 관행이다.

public long computeDurationInMilliseconds() {
  long duration = (((hours * 60) + minutes) * 60 + seconds ) * 1000 ; // bad
  return duration;
}

혹자는 무엇이 반환되는지에 대해 명시적으로 나타낼 수 있기 때문에 가독성이 향상될 수 있다고 주장한다.
하지만 이 변수는 메서드 호출자에게 노출되지 않는 정보이며, 무엇이 반환되는지에 대한 정보는 메서드명에서 유추할 수 있도록 하는 것이 올바르다고 볼 수 있다.

Twitter Style Guide: Superfluous temporary variables
SonarSource: Local variables should not be declared and then immediately returned or thrown

쓸모없는 평가​

특정 조건에만 평가되는 식은 실제로 사용될 때까지 평가를 늦춰야 한다.

logger.log(Level.DEBUG, "Something went wrong: " + message); // bad
logger.log(Level.DEBUG, "Something went wrong: {0} ", message); // good
logger.log(Level.DEBUG, () -> "Something went wrong: " + message); // good

SonarSource: "Preconditions" and logging arguments should not require evaluation

변수​

유효 범위​

변수는 최대한 사용되는 곳 근처에 선언하여 변수의 스코프를 최소화시켜야 한다.

String message = "message"; // bad
// do many things...
logger.info(message);

클린 코드: 수직 거리 (101page)
이펙티브 자바 3/E: 지역변수의 범위를 최소화하라 (344page)
SonarSource - Variables should not be declared before they are relevant

컬렉션​

자바 Collection의 목적은 구현 세부사항을 숨기기 위해 인터페이스를 노출시키는 것이다.
따라서 인스턴스화한 구현 클래스는 컬렉션 인터페이스에 할당되는 것이 이상적이다.

ArrayList<String> arrayList = new ArrayList<>(); // bad
List<String> list = new ArrayList<>(); // good

SonarSource: Declarations should use Java collection interfaces such as "List" rather than specific implementation classes such as "LinkedList"

원시 타입​

원시(raw) 타입은 컴파일 타임에 타입에 대한 안정성을 보장받을 수 없기 때문에 절대 사용하면 안된다.

List rawTypeList = new ArrayList(); // bad
List<String> genericList = new ArrayList<>(); // good

이펙티브 자바 3/E: 로 타입은 사용하지 말라 (154page)
Twitter Style Guide: Always use type parameters
SonarSource: Raw types should not be used

함수형 인터페이스​

간결한 코드를 위해 람다 표현식보다는 메서드 레퍼런스를 선호하도록 하자.

names.map(name -> name.toUpperCase())
    .forEach(name -> logger.log(name));

names.map(String::toUpperCase)
    .forEach(logger::log); // good

이펙티브 자바 3/E: 람다보다는 메서드 참조를 사용하라 (259page)
SonarSource: Lambdas should be replaced with method references

박싱 피하기​

Arrays.asList(T...)의 인자로 primitive 배열을 넘긴다면 박싱이 일어나게 된다.
반면 Arrays.stream(T[])은 가능한 경우 IntStream과 같은 primitive 스트림을 사용하려 하기 때문에 박싱이 일어나지 않는다.

Arrays.asList(1, 2, 3, 4).stream(); // bad
Arrays.stream(new int[] {1, 2, 3, 4}) // good

primitive 타입을 사용하는 함수형 인터페이스는 박싱이 일어날 수 밖에 없다.
자바는 오토 박싱을 피할 수 있도록 해당 제네릭 타입을 생략한 함수형 인터페이스도 제공한다.

Function<String, Integer> parser = Integer::parseInt; // bad
ToIntFunction<String> parser = Integer::parseInt; // good

SonarSource: "Arrays.stream" should be used for primitive arrays
SonarSource: Functional Interfaces should be as specialised as possible

null 피하기​

부주의한 null의 사용은 예상치 못한 버그를 발생시키는 원인이 된다.
null을 암묵적으로 동의하는 것보다는 빠르게 실패시키는 것이 개발자에게 더 도움이 될 것이다.

class AsyncFileReader {
  void readLater(File file, Closure<String> callback) {
    checkNotNull(file);
    checkArgument(file.exists() && file.canRead(), "File must exist and be readable.");
    checkNotNull(callback);
    ...
  }
}

특히 반환 값으로써의 null이 의미하는 것은 불쾌할 정도로 모호하다.
예를 들어, Map.get(key)는 맵에 key가 없는 경우에도 null을 반환할 수 있지만 실제 key에 담긴 value가 null일 수도 있다.

Guava: Using and avoiding null
Twitter Style Guide: Preconditions

문자열​

문자열 비교​

문자열 비교시 NullPointerException을 미연에 방지하기 위해 문자열 리터럴은 좌항에 두어야 한다.

if ("foo".equals(myString)) {
  ...
}

SonarSource: Strings literals should be placed on the left side when checking for equality

문자열 결합​

문자열은 기본적으로 불변하므로 더하기 연산자를 통해 서로 결합하는 행위는 새 문자열을 생성하는 것과 같다.
이러한 작업은 문자열이 길어질 수록 비효율적이므로 StringBuilder나 StringBuffer를 경우에 맞게 사용할 것을 권장한다.

StringBuilder stringBuilder = new StringBuilder();
for (int i = 0; i < arrayOfStrings.length; i++) {
  stringBuilder.append(arrayOfStrings[i]);
}
String string = stringBuilder.toString();

SonarSource: Strings should not be concatenated using '+' in a loop

문자열 변경​

String의 replaceAll(String) 메서드는 Pattern의 compile() 메서드를 이용하기 때문에 비용이 높다.
정규식을 사용하는 것이 아니라면 replace(String) 메서드로 대체할 것을 권장한다.

String init = "Bob is a Bird... Bob is a Plane... Bob is Superman!";
String changed = init.replace("Bob is", "It's");

이펙티브 자바 3/E: 문자열 연결은 느리니 주의하라 (366page)
SonarSource: "String#replace" should be preferred to "String#replaceAll"

컬렉션​

맵 순회​

맵에서 키만 필요한 경우 keySet()을 순회하는 것이 맞지만, 키와 값이 모두 필요한 경우에는 entrySet()을 순회하는 것이 더 효율적이다.

// bad
for (String key : map.keySet()) {
  Object value = map.get(key);
}

// good
for (Map.Entry<String, Object> entry : map.entrySet()) {
  String key = entry.getKey();
  Object value = entry.getValue();
}

SonarSource: "entrySet()" should be iterated when both the key and value are needed

열거형 컬렉션​

키가 열거형인 Map<K, V> 또는 Set<E>은 EnumMap<K, V>, EnumSet<E>으로 바꿔야 한다. 기본 데이터 구조가 단순 배열이기 때문에 기존 컬렉션보다 더 효율적이다.

SonarSource: Maps with keys that are enum values should be replaced with EnumMap
SonarSource: Sets with elements that are enum values should be replaced with EnumSet

동기화된 컬렉션​

자바 API의 초기 클래스들은 thread-safe하게 설계되어 오버헤드가 크다.
동시성을 제어하지 않아도 되는 환경이라면 다음과 같이 대체할 것을 권장한다.

  • Vector<T> → ArrayList<T>, LinkedList<T>
  • Stack<T> → Deque<T>
  • Hashtable<K, V> → HashMap<K, V>
  • StringBuffer → StringBuilder

Twitter Style Guide: List over Vector
SonarSource: Synchronized classes Vector, Hashtable, Stack and StringBuffer should not be used

예외​

예외 구체화​

메서드에서 Error, RuntimeException, Throwable, Excepion과 같은 일반적인 예외를 던져 버리면 호출하는 측에서는 이를 필터링할 수 있는 수단이 없다.

Twitter Style Guide: Catch narrow exceptions
SonarSource: Generic exceptions should never be thrown

표준 예외​

자바 라이브러리는 대부분 API에서 쓰기에 충분한 수의 표준 예외를 제공한다.
표준 예외는 이미 많은 프로그래머들에게 익숙해져 있기 때문에 코드를 읽기 쉽게 만들어준다는 장점이 있다.

표준 예외설명
IllegalArgumentException허용하지 않는 값이 인수로 건네졌을 때
(null은 따로 NullPointerException으로 처리)
IllegalStateException객체가 메서드를 수행하기에 적절하지 않은 상태일 때
NullPointerExceptionnull을 허용하지 않는 메서드에 null을 건넸을 때
IndexOutOfBoundException인덱스가 범위를 넘어섰을 때
ConcurrentModificationException허용하지 않는 동시 수정이 발견됐을 때
UnsupportedOperationException호출한 메서드를 지원하지 않을 때

이상이 가장 흔하게 사용되는 표준 예외지만, ArithmeticException이나 NumberFormatException처럼 좀 더 구체적인 상황을 묘사할 수 있는 예외도 존재한다.

커스텀 예외​

적절한 커스텀 예외는 사용자가 구현을 잘 알지 못하더라도 이를 올바르게 처리할 수 있게 유도할 수 있다.

Member findById(long id) throws MemberNotFoundException; // Member를 찾지 못했을 경우 MemberNotFoundException이 발생한다는 것을 쉽게 예상할 수 있다.

예외 단일화​

대다수의 상황에서 예외를 처리하는 방식은 예외를 일으킨 원인과는 무관하게 일정하다.

  1. 해당 오류를 기록한다.
  2. 프로그램을 계속 수행해도 좋은지 확인한다.

따라서 보통은 한 메서드 당 하나의 예외만 던져도 충분한 경우가 많다. 만약 레거시 라이브러리가 여러 예외를 던진다면 예외 타입 하나만을 반환하도록 래핑 클래스를 정의하는 방법을 고려해보자.

예외 위임하기​

예외 처리를 위임하려면 그 전에 먼저 해당 예외가 처리 가능한 예외인지 고려해봐야 한다.
스프링 프레임워크의 예외 처리 철학을 따라 처리가 불가능하다고 판단되는 CheckedException은 RuntimeException으로 래핑하여 던지자.

try {
  repository.find(Member.class, id);
} catch (SQLException exception) {
  throw new DataAccessException(exception);
}

예외 무시​

예외가 발생한 경우에는 적절한 조치를 취해주는 것이 일반적이다.
만약 아무것도 하지 않는 것이 적절한 경우에는 Exception의 변수명을 ignored 또는 expected로 선언한 뒤, 그에 대한 부연 설명을 주석으로 남겨주자.

try {
  emptyStack.pop();
} catch (NoSuchElementException ignored) {
  // just continue
}

오류 코드 패턴​

오류 코드를 반환하는 메서드는 호출자에게 오류 코드를 곧바로 처리해야 한다는 부담을 주며 여러 단계로 중첩되는 코드를 야기시킨다.

if (subscriptionRepository.removeAll(member) == ErrorCode.OK) {
  if (memberRepository.removeAll(member) == ErrorCode.OK) {
    logger.info("success");
  } else {
    logger.info("failed to remove member");
  }
} else {
  logger.info("failed to remove subscription");
}

오류 코드 대신 예외를 던지면 오류 처리 영역이 분리되므로 코드가 깔끔해진다.

try {
  subscriptionRepository.removeAll(member);
  memberRepository.remove(member);
} catch (DataAccessException exception) {
  logger.error(exception.getMessage());
}
