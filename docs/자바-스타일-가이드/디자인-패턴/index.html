디자인 패턴

디자인 패턴이란 가장 흔히 접할 수 있는 설계 문제들에 대한 보편적인 해결책이다.

Wikipedia: Software design pattern

생성 패턴​

생성 패턴은 객체를 생성하는 방법과 밀접한 관련이 있다.
제어된 방식으로 객체를 만듬으로써 복잡성과 불안정성을 줄인다.

Baeldung: Introduction to Creational Design Patterns
Wikipedia: Creational pattern

싱글톤 패턴​

싱글톤 패턴은 어플리케이션 전체에 걸쳐 단 하나의 인스턴스만 존재하도록 보장한다.
싱글톤 클래스는 항상 thread-safe 함을 보장하는 LazyHolder 기법을 사용해 구현하도록 하자.

public class Singleton {
  private Singleton() {}

  public static Singleton getInstance() {
    return SingletonHolder.INSTANCE;
  }

  private static class SingletonHolder {
    private static final Singleton INSTANCE = new Singleton();
  }
}

Baeldung: Double Checked Locking with Singleton

팩토리 메서드 패턴​

팩토리 메서드 패턴은 특정 종속성을 구현하기 위해 런타임 값이 필요한 경우 사용된다.
문자열 매개변수를 받아 분기하는 것 보다는 Enum을 활용하는 것을 권장한다.

public interface Animal { ... }
public class Dog implements Animal { ... }
public class Cat implements Animal { ... }

public enum AnimalType {
  DOG {
    @Override
    public Dog getInstance() {
      return new Dog();
    }
  },
  CAT {
    @Override
    public Cat getInstance() {
      return new Cat();
    }
  }

  public abstract Animal getInstance();
}

public class AnimalFactory {
  public Animal create(AnimalType animalType) {
    return animalType.getInstance();
  }
}

Wikipedia: Factory method pattern

추상 팩토리 패턴​

추상 팩토리 패턴은 팩토리 클래스 자체를 런타임에 결정한다.

public interface AbstractFactory { ... }
public class AnimalFactory implements AbstractFactory { ... }
public class PlantFactory implements AbstractFactory { ... }

public enum FactoryType {
  ANIMAL {
    @Override
    public AnimalFactory getInstance() {
      return new AnimalFactory();
    }
  },
  PLANT {
    @Override
    public PlantFactory getInstance() {
      return new PlantFactory();
    }
  }

  public abstract AbstractFactory getInstance();
}

public class FactoryProvider {
  public static AbstractFactory getFactory(FactoryType factoryType) {
    return factoryType.getInstance();
  }
}

Wikipedia: Abstract factory pattern

빌더 패턴​

생성자의 매개변수가 많다면 빌더 패턴을 고려해봐야 한다.
메서드 체이닝 방식으로 인스턴스를 생성할 수 있도록 해주어 코드가 깔끔해진다.

public class Member {
  private final String id;
  private final String name;

  private Member(String id, String name) {
    this.id = id;
    this.name = name;
  }

  public static class Builder {
    private String id;
    private String name;

    public Builder id(String id) {
      this.id = id;
      return this;
    }

    public Builder name(String name) {
      this.name = name;
      return this;
    }

    public Member build() {
      return new Member(this.id, this.name);
    }
  }

  // getter method
}

Baeldung: Builder pattern

구조 패턴​

구조 패턴은 클래스나 객체를 조합해 더 큰 구조를 만드는 패턴이다.
예를 들어 서로 다른 인터페이스를 지닌 2개의 객체를 묶어 단일 인터페이스를 제공하거나 객체들을 서로 묶어 새로운 기능을 제공한다.

프록시 패턴​

특정 객체를 호출하기 전후에 해야할 작업이 있다면 프록시 패턴을 사용할 수 있다.
다음은 초기화가 무거운 객체의 게으른 초기화를 프록시를 통해 구현하는 예이다.

public interface Printable {
  public void print();
}

public class Printer implements Printable {
  public Printer() {
    // very heavy initialization
  }

  @Override
  public void print() {
    System.out.println("hello, world");
  }
}

public class PrinterProxy implements Printable {
  private Printer printer;

  @Override
  public void print() {
    if (this.printer == null) {
      this.printer = new Printer();
    }
    this.printer.print();
  }
}

Baeldung: The Proxy Pattern in Java
Wikipedia: Proxy pattern

어댑터 패턴​

외부 컴포넌트가 현재 어플리케이션과 호환되지 않을 경우 적합한 어댑터를 개발하여 서로 호환되도록 할 수 있다.

public interface Formattable {
  public String format(String target);
}

public class Formatter implements Formattable {
  public String format(String target) {
    return target;
  }
}

public class FormatterAdapter implements Formattable {
  private final Formatter formatter;

  public FormatterAdapter(Formatter formatter) {
    this.formatter = formatter;
  }

  public String format(String target) {
    return "[" + this.formatter.format(target) + "]";
  }
}

Baeldung: The Adapter Pattern in Java
Wikipedia: Adapter pattern

데코레이터 패턴​

데코레이터 패턴은 주어진 상황 및 용도에 따라 객체에 책임을 덧붙이는 패턴으로, 기능 확장이 필요할 때 서브클래싱 대신 쓸 수 있는 유연한 대안이 될 수 있다.
InputStream이나 OutputStream에 부가 기능을 추가하고자 할 때 이 패턴을 사용해봤을 것이다.

public interface Decoratable {
  public String getDescription();
}

public class ChristmasTree implements Decoratable {
  @Override
  public String getDescription() {
    return "christmas tree"
  }
}

public abstract class Decorator implements Decoratable {
  protected Decoratable decorateTarget;

  public Decorator(Decoratable decorateTarget) {
    this.decorateTarget = decorateTarget;
  }
}

public class Bulb extends Decorator {
  @Override
  public String getDescription() {
    return this.decorateTarget.getDescription() + " with bulb";
  }
}

Wikipedia: Decorator pattern

컴포지트 패턴​

컴포지트 패턴은 객체 간의 관계를 트리 구조로 구성하여 부분-전체 계층을 표현한다.

public interface Node {
  public void print();
}

public class File implements Node {
  @Override
  public void print() {
    System.out.println("file");
  }
}

public class Directory implements Node {
  private List<File> files = new ArrayList<>();

  public void add(File file) {
    files.add(file);
  }

  @Override
  public void print() {
    files.forEach(File::print);
  }
}

Wikipedia: Composite pattern

경량 패턴​

경량 패턴은 인스턴스화가 비싸거나 그 횟수가 빈번한 경우 객체를 재활용할 수 있는 방법을 제공한다.
경량 패턴을 적용할 객체는 반드시 불변해야 한다.

public class Vehicle { ... }

public class VehicleFactory {
  private static Map<Color, Vehicle> vehicleCache = new HashMap<>();

  public static Vehicle create(Color color) {
    return vehicleCache.computeIfAbsent(color, newColor -> {
      return new Vehicle(newColor);
    });
  }
}

Wikipedia: Flyweight pattern

커맨드 패턴​

커맨드 패턴은 요청을 객체의 형태로 표현하여 추후에 처리할 수 있는 방법을 제공한다.

public class Macro {
  private final List<Runnable> actions = new ArrayList<>();

  public void addAction(Runnable action) {
    actions.add(action);
  }

  public void run() {
    actions.forEach(Runnable::run);
  }
}

옵저버 패턴​

객체의 상태 변화를 관찰하고자 할 때 옵저버 패턴을 사용할 수 있다.
구현 시 PropertyChangeListener와 PropertyChangeSupport를 사용하면 용이하다.

public class Publisher {
  private final PropertyChangeSupport support;

  public Publisher() {
    this.support = new PropertyChangeSupport(this);
  }

  public void addPropertyChangeListener(PropertyChangeListener listener) {
    support.addPropertyChangeListener(listener);
  }

  public void removePropertyChangeListener(PropertyChangeListener listener) {
    support.removePropertyChangeListener(listener);
  }

  public void publish() {
    System.out.println("publish");
  }
}

public class Observer implements PropertyChangeListener {
  @Override
  public void propertyChange(PropertyChangeEvent event) {
    System.out.println("subscribe");
  }
}

만약 jdk1.9 이상을 사용한다면 Flow API를 사용할 것을 권장한다.

Wikipedia: Observer pattern

전략 패턴​

전략 패턴을 이용하면 알고리즘을 런타임에 결정하여 실행할 수 있다.

public enum DiscountStrategy {
  NORMAL(price -> price),
  HAPPY_HOUR(price -> price / 2);

  private IntUnaryOperator discounter;

  private DiscountStrategy(IntUnaryOperator discounter) {
    this.discounter = discounter;
  }

  public int discount(int price) {
    return this.discounter.apply(price);
  }
}

Wikipedia: Strategy pattern

Null 객체 패턴​

Null 객체 패턴의 의의는 null 대신 Null 객체를 사용함으로써 null 확인을 최소화함에 있다.
하지만 오류, 버그가 정상적인 흐름처럼 보일 수 있기 때문에 조심해서 사용해야 하는 패턴이기도 하다.

public interface Router {
  void route(Message msg);
}

public class SmsRouter implements Router {
  @Override
  public void route(Message msg) {
    // implementation details
  }
}

public class NullRouter implements Router {
  @Override
  public void route(Message msg) {
    // do nothing
  }
}
