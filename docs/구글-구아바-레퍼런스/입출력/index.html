입출력

파일 입출력 작업을 단순화시킨 guava의 파일 유틸리티에 대해 설명한다.

ByteStreams와 CharStreams​

stream이란 리소스의 위치 상태를 가지고 있는 I/O 데이터에 대한 Closeable 스트림을 나타낸다.
byte stream이란 InputStream 또는 OutputStream을 나타낸다.
char stream이란 Reader 또는 Writer를 나타낸다.

byte stream과 char stream은 각각 ByteStreams와 CharStreams 유틸리티 클래스에 대응된다.

byte[] toByteArray(InputStream)String toString(Readable)
N/AList<String> readLines(Readable)
long copy(InputStream, OutputStream)long copy(Readable, Appendable)
void readFully(InputStream, byte[])N/A
void skipFully(InputStream, long)void skipFully(Reader, long)
OutputStream nullOutputStream()Writer nullWriter()

guava의 스트림 관련 유틸리티들은 효율성을 위해 한 번에 전체 스트림을 처리하고 자체적으로 버퍼링을 제공한다.
다만 스트림을 닫는 것은 스트림을 연 코드의 책임이기 때문에 자동으로 닫아주지는 않는다.

Source와 Sink​

사용자가 스트림을 직접 처리하지 않도록 해주는 것은 입출력 유틸리티의 기본이다.
그 예시로 Files에는 toByteArray(File)와 write(File, byte[])가 있다.

접근하는 방식에 따라 유사한 메서드들이 여러군데 흩어져있기도 하다.
Resources의 toByteArray(URL)는 File 대신 URL을 사용하여 접근한다.

구아바는 이렇게 유사한 메서드들이 산발해있는 문제를 해결하기 위해 데이터 유형들을 Source와 Sink로 추상화하였다.
Source와 Sink는 File이나 URL처럼 새로운 스트림을 여는 방법을 알고 있는 일종의 리소스다.

  • Source: 읽을 수 있는 리소스다.
  • Sink: 쓸 수 있는 리소스다.

그리고 Source와 Sink는 처리하고자 하는 것이 byte인지 char인지에 따라 구현이 분리되어 있다.

읽기ByteSourceCharSource
쓰기ByteSinkCharSink

또한 Source와 Sink 클래스를 직접 확장하여 새로운 구현을 만들 수도 있다.

그러나 열려있는 스트림을 래핑하는 Source나 Sink를 만들어서는 안된다.
openStream() 메서드가 호출될 때마다 새로운 스트림을 여는 것이 바람직하다.

Guava: Creating sources and sinks

Source와 Sink의 사용​

모든 Source 및 Sink는 읽고 쓰기 위해 새로운 스트림을 열 수 있다.

  • openStream(): Source인지 Sink인지에 따라 InputStream, Reader 또는 OutputStream, Writer가 반환된다.
  • openBufferedStream(): Source인지 Sink인지에 따라 InputStream, Reader 또는 OutputStream, Writer가 반환된다는 것은 같지만, 반환된 스트림이 필요한 경우 버퍼링이 될 수 있음을 보장한다.

사용 예​

// Read the lines of a UTF-8 text file
ImmutableList<String> lines = Files.asCharSource(file, Charsets.UTF_8)
    .readLines();

// Count distinct word occurrences in a file
Multiset<String> wordOccurrences = HashMultiset.create(
    Splitter.on(CharMatcher.whitespace())
        .trimResults()
        .omitEmptyStrings()
        .split(Files.asCharSource(file, Charsets.UTF_8).read()));

// SHA-1 a file
HashCode hash = Files.asByteSource(file).hash(Hashing.sha1());

// Copy the data from a URL to a file
Resources.asByteSource(url).copyTo(Files.asByteSink(file));

리소스 해제​

Closeable 리소스를 사용한 후 제대로 닫는 일은 소켓 연결을 끊고, 파일 디스크립터가 누출되지 않도록 하고, 프로그램이 올바른 동작을 하도록 보장하는데 중요하다.

JDK7 이전에는 이를 제대로 지키기가 매우 어려웠으며, JDK 내에서 조차 올바르게 수행되지 못한 것으로 악명높았다.
다음은 JDK6에서 InputStream을 열고 닫는 코드다.

InputStream in = null;
try {
  in = openInputStream();
  // do something with in
} finally {
  if (in != null) {
    in.close();
  }
}

위 코드는 복잡할 뿐더러 심각한 문제를 내포하고 있다.
만약 try 블록에서 예외가 발생한 뒤 finally 블록의 in.close()에서도 예외가 발생한다면 try 블록에서 발생한 예외가 finally 블록에서 발생한 예외에 의해 삼켜질 것이고 개발자는 그 예외의 세부사항을 보지 못할 뿐더러 발생했다는 사실 조차 눈치채지 못할 것이다.

동시에 여러 개의 스트림을 열어야 한다면 상황은 더 나빠질 수 밖에 없다.

InputStream in = null;
try {
  in = openInputStream();
  OutputStream out = null;
  try {
    out = openOutputStream();
    // do something with in and out
  } finally {
    if (out != null) {
      out.close();
    }
  }
} finally {
  if (in != null) {
    in.close();
  }
}

InputStream vs OutputStream​

리소스를 닫을 때 InputStream과 OutputStream 사이에는 중요한 차이가 있다.

InputStream을 닫으려고 시도할 때 발생하는 예외는 프로그램 입장에서 그리 중요하지 않을 수도 있다.
닫는 시점에서 InputStream은 모두 읽힌 상태이기 때문이다.

반면 OutputStream을 닫으려고 시도할 때 발생하는 예외는 중요하게 처리되어야 한다.
OutputStream이 데이터를 버퍼링하고 있을 수 있어 close()가 호출될 때 버퍼링된 데이터를 플러시해야 하기 때문이다.
다시 말해, close()가 호출될 때 발생한 예외는 의도된 쓰기 작업을 완료하지 못한 체 실패했음을 나타낸다.

해결책​

try-with-resources​

자바 7 이상을 사용하는 경우 try-catch-resources 문을 사용하여 try 문이 끝날 때 자동으로 스트림을 닫을 수 있다.

try (InputStream in = openInputStream();
     OutputStream out = openOutputStream()) {
  // do stuff with in and out
}

만약 예외가 발생한다면 스트림을 자동으로 닫은 후 try 문을 끝낼 것이기 때문에 JDK7 이전의 방식처럼 예외가 다른 예외에 의해 삼켜지는 상황은 발생하지 않는다.

Source와 Sink​

ByteSource와 CharSource는 읽을 수 있는 바이너리 데이터와 문자 데이터를 나타낸다.
ByteSink와 ByteSource는 쓸 수 있는 바이너리 데이터와 문자 데이터를 나타낸다.

Source나 Sink를 사용하면 스트림을 자동으로 닫아주기 때문에 해당 문제를 완전히 회피할 수 있다.
다음은 Files를 이용하여 간단하게 File을 ByteSink로 복사하는 예다.

ByteSink sink = ...
Files.asByteSource(file).copyTo(sink);
