Optional API

Optional<T>을 이용하면 객체가 null일 경우에 대한 처리를 보다 유연하게 만들 수 있다.

생성 메서드​

of​

객체가 null이 아니라는 가정하에 Optional<T>를 생성한다.
만약 T가 null이라면 NullPointerException이 발생한다.

Optional<String> maybeName = Optional.of("scott");

ofNullable​

객체가 null일수도 있다는 가정하에 Optional<T>를 생성한다.

Optional<String> maybeName = Optional.ofNullable("scott");

empty​

비어있는 Optional<T>를 생성한다.

Optional<String> maybeName = Optional.empty();

primitive 타입용 Optional​

Optional<T>에 primitive 타입을 넣을 수는 있지만 auto boxing이 일어날 수 밖에 없다.

Optional<Integer> maybeInt = Optional.of(1);

그래서 primitive 타입을 위한 OptionalInt, OptionalLong, OptionalDouble을 따로 제공하고 있다.

OptionalInt maybeInt = OptionalInt.of(1);
OptionalLong maybeLong = OptionalLong.of(2);
OptionalDouble maybeDouble = OptionalDouble.of(3);

조회 메서드​

isPresent​

Optional<T>에 담긴 T가 null이 아닌지 확인한다.

if (maybeName.isPresent()) {
  System.out.println("present");
}

get​

Optional<T>에 담긴 T를 반환한다.

String name = maybeName.get();

orElse​

Optional<T>에 담긴 T를 반환한다.
T가 null이라면 인자로 넘긴 디폴트 값을 반환한다.

String name = maybeName.orElse("default");

orElseGet​

Optional<T>에 담긴 T를 반환한다.
T가 null이라면 인자로 넘긴 Supplier<T>의 반환 값을 그대로 반환한다.

String name = maybeName.orElseGet(() -> "default");

orElseThrow​

Optional<T>에 담긴 T를 반환한다.
T가 null이라면 NoSuchElementException이 발생한다.

String name = maybeName.orElseThrow();

또는 특정 Exception을 지정할 수도 있다.

String name = maybeName.orElseThrow(IllegalArgumentException::new);

연산 메서드​

filter​

Predicate<T>의 반환 값이 true인 경우에만 Optional<T>을 반환하고, 아닌 경우에는 빈 Optional<T>을 반환한다.

String name = maybeName.filter(name -> name.startsWith("s")).get();

map​

Function<T, R>으로 T를 R로 변환한다.

String name = maybeName.map(name -> "hello " + name).get();

flatMap​

Function<T, R>의 반환 값이 Optional<R>인 경우 R로 flat하게 변경하고자 할 때 사용된다.

String name = maybeName.flatMap(name -> Optional.ofNullable(name)).get();

권고사항​

Optional<T>는 다음과 같은 이유로 메서드의 반환 값으로만 사용하기를 권장하고 있다.

  • Optional을 메서드의 매개변수로 사용하는 경우
    메서드의 매개변수로 null이 넘어오는 경우와 Optional<T>이 비어있는 경우를 모두 고려해야 하기 때문에 깊은 의심(deep doubt) 패턴이 일어날 수 밖에 없다.
    Optional<T>를 사용하지 않을 때보다 더 번거로워질 것이다.
  • Optional을 Map의 키로 사용하는 경우
    Map<K, V>의 특징 중 하나는 K가 null이 될 수 없다는 것이다.
    비어있는 Optional<T>을 키로 사용하는 것은 이러한 제약을 깨트리는 행위다.
  • Optional을 클래스의 필드로 사용하는 경우
    어떤 클래스의 필드가 있을 수도 있고 없을 수도 있다는 것은 도메인 설계상 좋지 못하다.
    보편적인 해결방식은 해당 필드가 없는 상위 클래스와 필드가 있는 하위 클래스로 나누는 것이다.
  • Optional을 반환하는 메서드에서 null을 반환하는 경우
    이는 메서드를 호출하는 클래스에서 깊은 의심(deep doubt)을 하게 만든다.
    빈 값을 반환해야 하는 경우에는 Optional.empty()를 사용하는 것이 올바르다.
  • Optional에 컨테이너성 타입을 넣는 경우
    Collection<E>, Map<K, V>, Stream<T> 등 컨테이너성 타입은 이미 자체적으로 비어있는 상태를 표현할 수 있다.
    여기에 Optional<T>을 사용하는 것은 두 번 감싸는 것과 마찬가지이므로 의미가 없다.
