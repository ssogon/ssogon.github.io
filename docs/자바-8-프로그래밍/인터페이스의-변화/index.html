인터페이스의 변화

자바 8부터는 인터페이스에 abstract 메서드 뿐만 아니라 default, static 메서드도 가질 수 있게 되었다.

default 메서드​

default 메서드는 인터페이스를 구현할 인스턴스들이 공통적으로 사용할 메서드를 정의한다.
요점은 인터페이스에서 메서드를 '구현'할 수 있다는 것이다. 마치 추상 클래스와 유사해 보인다.

public interface Foo {
  default void printHello() {
    System.out.println("hello");
  }
}

Iterable의 default 메서드​

Iterable::forEach(Consumer<T>) 메서드는 각 요소들을 차례대로 순회한다.

Iterable<String> names = Arrays.asList("scott", "michael", "holden");
names.forEach(System.out::println);

Iterable::spliterator() 메서드는 반으로 쪼갤 수 있는 기능을 가진 Spliterator<T>를 반환한다.
추후 설명할 Stream<T>의 병렬 파이프라이닝 처리를 위해 고안된 것으로, 직접 사용하는 경우는 드물다.

Iterable<String> names = Arrays.asList("scott", "michael", "holden");
Spliterator<String> nameSpliterator1 = names.spliterator();
Spliterator<String> nameSpliterator2 = nameSpliterator1.trySplit();
nameSpliterator1.forEachRemaining(System.out::println); // michael holden
nameSpliterator2.forEachRemaining(System.out::println); // scott

Spliterator<T>에 대한 더 자세한 내용은 아래 링크를 참고바란다.

Collection의 default 메서드​

Collection::stream() 메서드는 각 요소들을 functional하게 처리할 수 있는 Stream<T> 객체를 반환한다.
Stream<T>에 관한 내용은 스트림 API에서 다루겠다.

List<String> names = Arrays.asList("scott", "michael", "holden");
names.stream()
  .map(String::toUpperCase)
  .forEach(System.out::println);

Collection::removeIf(Predicate<T>)는 Predicate<T>의 결과가 참인 요소를 제거한다.

List<String> names = Arrays.asList("scott", "michael", "holden");
names.removeIf(name -> name.startsWith("s"));
names.forEach(System.out::println); // michael holden

Comparator의 default 메서드​

Comparator::reversed()는 정렬 순서를 반전시킨 Comparator<T>를 반환한다.

List<String> words = Arrays.asList("apple", "dog", "cat");
Comparator<String> comparator = Comparator.naturalOrder();
words.sort(comparator.reversed());
words.forEach(System.out::println); // dog cat apple

static 메서드​

static 메서드는 헬퍼 또는 유틸리티를 제공하고자 할 때 유용하다.
익히들 아는 클래스의 static 메서드와 동일하다. 인터페이스에서도 가질 수 있게 된 것 뿐이다.

public interface Foo {
  static void printHello() {
    System.out.println("hello");
  }
}

추상 클래스와의 비교​

자바 8에 들어서 인터페이스와 추상 클래스의 경계가 조금 더 모호해졌다.
인터페이스에 default 메서드가 추가되면서 '틀' 뿐만 아니라 '구현' 또한 가질 수 있게 되었기 때문이다.

다음 중 해당하는 사항이 있다면 추상 클래스 사용을 고려해보자.

  • 밀접하게 관련된 여러 클래스들 간에 코드를 공유하고 싶은 경우
  • 상속받을 클래스들이 공통으로 가지는 필드와 메서드가 많거나 public 이외의 접근제어자가 필요한 경우
  • static이 아니거나 final이 아닌 필드를 선언하고 싶은 경우

다음 중 해당하는 사항이 있다면 인터페이스 사용을 고려해보자.

  • 서로 관련성이 없는 클래스들 간에 공통 행동을 명시하고 싶은 경우
    (예를 들어, Comparable이나 Cloneable의 구현 클래스들은 관련성이 없다.)
  • 다중 상속의 이점을 활용하고 싶은 경우

추상 클래스는 is-a 관계, 인터페이스는 has-a 관계를 표현한다고 볼 수도 있겠다.
