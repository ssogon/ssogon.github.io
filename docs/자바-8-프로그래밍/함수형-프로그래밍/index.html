함수형 프로그래밍

함수를 변수처럼 다루는 언어를 두고 일급 함수를 가졌다고 말한다.
일급 함수를 가진 대표적인 언어로는 자바스크립트가 있다.

// 자바스크립트의 예
const echo = function(message) {
  console.log(message);
}
echo("hello, world");

반면 자바는 메서드를 하나만 가지고 있는 객체를 마치 함수인 것처럼 취급하여 함수형 프로그래밍을 지원한다.

// 자바의 예
Consumer<String> echo = new Consumer<String>() {
  @Override
  public void accept(String message) {
    System.out.println(message);
  }
};
echo.accept("hello, world");

익명 클래스​

함수형 인터페이스를 알아보기 전에 그 근간이 되는 익명 클래스를 먼저 알아야 한다.
익명 클래스란 class 키워드를 사용하지 않고 다른 클래스를 확장하는 방식으로 선언 및 인스턴스화가 이루어지는 특수한 클래스를 말한다. 이는 자바 8 이전부터 있던 기능이다.
이때 클래스명을 별도로 지정하지 않기 때문에 익명 클래스라 부른다.

클래스를 상속하여 만들기​

다음은 기존 클래스를 상속하여 익명 클래스를 생성하는 예이다.

// class 익명 extends Dog
Dog retriever = new Dog() {
  @Override
  public String getName() {
    return "retriever";
  }
}

인터페이스를 구현하여 만들기​

다음은 기존 인터페이스를 구현하여 익명 클래스를 생성하는 예이다.

// class 익명 implements Runnable
Runnable runner = new Runnable() {
  @Override
  public void run() {
    logger.info("run");
  }
}

함수형 인터페이스​

함수형 인터페이스란 추상 메서드를 딱 하나만 가지고 있는 인터페이스를 말한다.

public interface Hashable {
  public byte[] hash(byte[] bytes); // abstract 키워드는 생략 가능
}

static 메서드나 default 메서드는 몇 개가 있더라도 상관없다.
중요한 것은 추상 메서드가 단 하나만 존재하는가이다.

public interface Hashable {
  public byte[] hash(byte[] bytes);
  public static void doNothing() { ... }
  public default void doSomething() { ... }
}

함수형 인터페이스에 @FunctionalInterface 어노테이션을 달아주면 함수형 인터페이스의 조건을 위반했을 시 컴파일 에러를 발생시켜주기 때문에 오류를 보다 빠르게 발견할 수 있다.

@FunctionalInterface
public interface Hashable {
  public byte[] hash(byte[] bytes);
  public byte[] doNothing(); // compile error
}

built-in 함수형 인터페이스​

함수형 인터페이스는 인자 여부와 반환 여부에 따라 크게 네 가지로 나뉠 수 있다.

  • 실행만 하는 함수
  • 인자 T를 받는 함수
  • 값 T를 반환하는 함수
  • 인자 T를 받고, 값 R을 반환하는 함수

자바는 이렇게 자주 사용될만한 함수형 인터페이스들을 java.lang.function 패키지에 미리 정의해두었다.
여러 경우를 상정해두었기 때문에 함수형 인터페이스를 직접 정의해서 사용하는 일은 드물 것이다.

함수형 인터페이스구현해야할 메서드설명
Runnablevoid run()실행한다.
Callable<T>T call()실행 후, 결과 값 T를 리턴한다.
Comsumer<T>void accept(T)T를 인자로 받아 소비한다.
BiConsumer<T, U>void accept(T, U)T, U를 인자로 받아 소비한다.
Supplier<T>T get()T를 제공한다.
Function<T, R>R apply(T)T를 인자로 받아 R을 리턴한다.
BiFunction<T, U, R>R apply(T, U)T, U를 인자로 받아 R을 리턴한다.
UnaryOperator<T>T apply(T)T를 인자로 받아 T를 리턴한다.
BinaryOperator<T>T apply(T, T)T, T를 인자로 받아 T를 리턴한다.
Comparator<T>int compare(T, T)T, T를 인자로 받아 int를 리턴한다.
Predicate<T>boolean test(T)T를 인자로 받아 boolean을 리턴한다.
BiPredicate<T, U>boolean test(T, U)T, U를 인자로 받아 boolean을 리턴한다.

이 외에도 다양한 함수형 인터페이스들이 미리 준비되어 있다.

람다 표현식​

함수형 인터페이스를 구현한 익명 클래스는 람다식이라는 형태로 표현될 수 있다.

// 익명 클래스를 이용하는 방식
Hashable sha256Hasher = new Hashable() {
  @Override
  public byte[] hash(byte[] bytes) {
    return Hashing.sha256().hashBytes(bytes).asBytes();
  }
};
// 람다 표현식을 사용한 방식
Hashable sha256Hasher = (bytes) -> Hashing.sha256().hashBytes(bytes).asBytes();

람다 표현식을 사용했을 때 메서드의 구현을 제외하고 모든 것이 생략된 것을 볼 수 있다.

이와 같이 비약적인 생략이 가능해진 이유는 함수형 인터페이스가 추상 메서드를 단 하나만 가지고 있다는 대전제 덕분이다.
구현해야할 메서드가 단 하나이니 메서드명을 명시하지 않아도 컴파일러가 충분히 유추할 수 있는 것이다.

포맷​

람다식은 기본적으로 (인자) -> {바디} 형식으로 작성된다.

Consumer<String> printMessage = (String message) -> { System.out.println(message); };
Supplier<String> getMessage = () -> { return "message"; };

인자의 타입은 컴파일러가 제네릭 타입을 보고 추론할 수 있기 때문에 생략할 수 있다.

Consumer<String> printMessage = (message) -> { System.out.println(message); };

인자가 하나라면 소괄호도 생략할 수 있다.

Consumer<String> printMessage = message -> { System.out.println(message); };

바디가 한 줄인 경우엔 중괄호 또한 생략할 수 있다.

Consumer<String> printMessage = message -> System.out.println(message);

바디가 한 줄이며 반환 값이 있는 경우 return도 생략 가능하다.

Supplier<String> getMessage = () -> "message";

이것들을 처음부터 굳이 외우려고 하지 않아도 좋다.
최신 IDE를 사용한다면 어떤 식으로 작성하더라도 가장 간결화된 방법을 제시해줄 것이다.

캡처링​

람다 표현식은 선언 후 어디서든 호출될 수 있기 때문에 람다식 밖의 외부 변수를 사용하는 경우 그 변수의 복사본을 만들어 둔다.
이를 캡처링(capturing)이라고 한다.

public class SupplierProvider {
  public static Supplier<String> getSupplier() {
    String localVariable = "local variable";
    return () -> localVariable;
  }
}
Supplier<String> supplier = SupplierProvider.getSupplier();
logger.info(supplier.get()); // "local variable"

변수화된 람다식은 다른 스레드에서 사용될 여지가 있으므로 캡처링된 변수는 동시성 문제에서 벗어날 수 없다.
자바는 비용을 들여 캡처링된 변수의 동기를 맞추기 보다는 이를 회피하기 위해 캡처링될 변수는 반드시 final 또는 effectively final이 되도록 제한하고 있다. (여기서 effectively final이란 선언 후 단 한 번도 변경되지 않아 마치 상수처럼 취급되는 변수를 말한다.)

int index = 0;
Runnable runner = () -> index++; // compile error

다만 객체의 상태를 변경하는 행위는 감지하지 못한다는 허점이 있다.

Member member = new Member();
Runnable runner = () -> member.setName("scott");

쉐도잉​

람다 표현식은 익명 클래스와 달리 쉐도잉(shadowing)이 일어나지 않는다.
즉, 람다를 감싸고 있는 스코프와 동일한 스코프를 가진다.

int index = 0;
Runnable runner = () -> {
  int index = 0; // compile error
};

메서드 레퍼런스​

메서드 레퍼런스는 람다 표현식을 보다 간결하게 표현하는 방법이다.
컴파일러가 충분히 유추할 수 있다고 판단되는 것들을 좀 더 줄여보겠다는 취지다.

static 메서드 레퍼런스​

args -> Class.method(args) 형태의 람다식을 메서드 레퍼런스로 변환하는 예이다.

// 람다 표현식을 사용하는 방식
LongSupplier epochTimeSupplier = () -> System.currentTimeMillis();
long epochTime = epochTimeSupplier.getAsLong();
// 메서드 레퍼런스를 활용한 방식
LongSupplier epochTimeSupplier = System::currentTimeMillis;
long epochTime = epochTimeSupplier.getAsLong();

특정 객체의 메서드 레퍼런스​

args -> instance.method(args) 형태의 람다식을 메서드 레퍼런스로 변환하는 예이다.

// 람다 표현식을 사용하는 방식
Consumer<String> printer = message -> System.out.println(message);
printer.accept("message");
// 메서드 레퍼런스를 활용한 방식
Consumer<String> printer = System.out::println;
printer.accept("message");

임의 객체의 메서드 레퍼런스​

(instance, args) -> instance.method(args) 형태의 람다식을 메서드 레퍼런스로 변환하는 예이다.

// 람다 표현식을 사용하는 방식
UnaryOperator<String> upperOperator = target -> target.toUpperCase();
String upperName = upperOperator.apply("scott");
// 메서드 레퍼런스를 활용한 방식
UnaryOperator<String> upperOperator = String::toUpperCase;
String upperName = upperOperator.apply("scott");

생성자 레퍼런스​

(args) -> new Class(args) 형태의 람다식을 메서드 레퍼런스로 변환하는 예이다.

// 람다 표현식을 사용하는 방식
Supplier<Member> memberSupplier = () -> new Member();
Member member = memberSupplier.get();
// 메서드 레퍼런스를 활용한 방식
Supplier<Member> memberSupplier = Member::new;
Member member = memberSupplier.get();
